# 移动策略

MovementExecutor 负责消费移动需求（MovementDemand）并驱动角色移动。移动需求由两个正交维度描述：

- **DemandType** — 怎么移动（MoveTo 寻路 / TP 瞬移 / Hold 停留）
- **MovementPolicy** — 什么时候移动（Mechanic 惰性 / Gather 尽快）

## Mechanic — 惰性策略

**所有职业都尽可能晚地移动，只在必须走的时候才走。**

非读条职业（Tank / Melee / Physical Ranged）在 `deadline - travelTime` 时出发，卡着 deadline 到达。

读条职业（Caster / Healer）在此基础上考虑读条，每帧做一次前向计算：

```
oneMoreGcd = 当前时间 + GCD剩余 + GCD时长 + 移动耗时

还能塞一个 GCD 读条吗？
  oneMoreGcd ≤ deadline  →  等待，继续打
塞不下了，看当前状态：
  正在读条 + 滑步窗口来得及  →  等读条到滑步窗口再走（不浪费当前读条）
  正在读条 + 滑步也来不及   →  打断读条立刻走
  没在读条                  →  立刻走（放弃这个 GCD 机会）
```

为什么用 GCD 剩余而不是读条剩余？因为 GCD 冷却和读条是并行的。一个 2.5s 读条的 GCD 冷却也是 2.5s，读条结束时 GCD 刚好转好。用 GCD 剩余做前瞻天然覆盖了"当前读条结束到下一个 GCD 可用"这段时间。

## Gather — 集合策略

**尽快到达目标位置。**

非读条职业立刻走。读条职业看当前状态：
- 没在读条 → 立刻走
- 正在读条 → 等滑步窗口（读条剩余 ≤ 450ms）再走，不浪费当前技能

## 组合矩阵

| | Mechanic（惰性） | Gather（尽快） |
|---|---|---|
| **MoveTo** | 卡 deadline 走（读条职业有前向策略） | 立刻走（读条职业等滑步） |
| **TP** | 卡 deadline 瞬移（移动耗时=0） | 立刻瞬移 |
| **Hold** | 卡 deadline 停留 | 立刻停留 |

## 使用方式

```csharp
// 机制策略（默认）— 处理 Boss 机制，惰性移动
var demand = new MovementDemand
{
    Type = DemandType.MoveTo,
    TargetPos = targetPos,
    Policy = MovementPolicy.Mechanic,  // 默认值，可省略
};

// 集合策略 — 站位调整、集合分摊，尽快到达
var demand = new MovementDemand
{
    Type = DemandType.MoveTo,
    TargetPos = targetPos,
    Policy = MovementPolicy.Gather,
};
```
