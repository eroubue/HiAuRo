# HiAuRo 自动副本攻略 — 它是怎么帮你打通整个副本的？

## 一句话总结

HiAuRo 不只是帮你按技能——它的终极目标是让你的角色在副本里**全自动移动、全自动输出、全自动治疗减伤**，像一个真正会打副本的玩家一样。

---

## 核心思路：三个问题，三层回答

打副本本质上只有三件事：

```
① Boss 接下来要做什么？     → 事实轴回答
② 我们应该怎么应对？        → 决策引擎回答
③ 具体到我，该怎么做？       → 智能层 + ACR 回答
```

这三层各司其职，合在一起就是完整的自动攻略。

## 第一层：事实轴 — 看懂 Boss

事实轴是整个系统的"眼睛"。它持有一份 Boss 的技能时间表，知道：

- 现在是第几阶段（P1、P2、P3...）
- Boss 下一个技能是什么、还有多久放
- 每个技能的机制是什么（全屏 AOE？点名？需要分散？需要集合？）
- 安全区域在哪里（坐标）

这份时间表是提前制作好的 JSON 文件，按副本编号存放。进入副本时自动加载。

### 时间表是怎么推进的？

```
进入副本 → 加载时间表 → 战斗开始
  ↓
按时间推进：第 8 秒应该有 Boss 技能 A
  ↓
实际游戏中 Boss 确实在第 8 秒读条了 → 时间校准成功，偏差修正
  ↓
继续推进下一个事件...
  ↓
如果 Boss 血量触发了阶段切换 → 跳转到对应分支
```

关键设计：时间表不是死板地按秒数走，它会**校准**。如果实际 Boss 动作比预期早了或晚了半秒，系统会自动修正后续所有时间点。

## 第二层：决策引擎 — 做出应对方案

事实轴说"Boss 马上要放全屏大招了"，决策引擎负责决定"我们怎么扛过去"。

### 减伤分配

```
事实轴：Boss 在第 30 秒放全屏 AOE，需要 20% 减伤
  ↓
决策引擎检查队伍：
  - 骑士有"壁垒"（减伤 15%，冷却 90 秒）→ 可用 ✓
  - 学者有"鼓舞"（减伤 10%，冷却 60 秒）→ 可用 ✓
  - 白魔有"庇护所"（减伤 0%，是治疗不是减伤）→ 跳过
  ↓
分配：骑士壁垒(15%) + 学者鼓舞(10%) = 25% ≥ 20% → 够了！
  ↓
通知对应职业的 ACR 在指定时间释放技能
```

分配原则：**冷却短的优先用**。这样下一次需要减伤时，它们更可能转好。

### 治疗分配

类似减伤，但更紧急——治疗需求会**立即执行**，不等 deadline。

### 站位需求

这是最关键的部分。事实轴的每个事件可以携带**站位需求**：

```
Boss 即将放十字形 AOE → 事实轴说："需要在第 28 秒前到达安全位置"
  ↓
辅助轴脚本节点：根据 Boss 朝向 + 你的职业 + 队伍站位
  → 计算出你的安全坐标：东北角 (10, 0, -15)
  → 生成移动需求，写入缓冲区
  ↓
智能层：监控到这个事件到了 → 释放移动需求
  ↓
移动执行器：驱动角色走到 (10, 0, -15)
```

## 第三层：智能层 + ACR — 具体执行

### 辅助轴：计算安全坐标

辅助轴的脚本节点负责计算安全坐标。每个副本的 Boss 机制不同，安全位置的算法也不同——有的是固定坐标，有的要根据 Boss 朝向动态计算，有的要根据队伍人数分散站位。这些计算逻辑写在辅助轴的脚本节点里，按副本编号加载。

```
辅助轴脚本节点（每帧运行）
  → 读取当前 Boss 状态 / 队伍信息
  → 计算安全坐标
  → 生成移动需求，写入缓冲区
```

### 智能层：等待时机，释放需求

智能层不计算坐标，它做的是**协调**：

1. **监控事实轴** — 等事实轴推进到某个事件（比如"Boss 读条全屏 AOE"）
2. **释放移动需求** — 把辅助轴已经算好的、绑定到这个事件的移动需求释放出来
3. **交给移动执行器执行**

#### 两种移动策略

**惰性策略（打 Boss 机制时用）**：尽量晚走，多打一个技能再走。

```
读条职业的决策过程：
  → 还能塞一个读条技能吗？
  → 能 → 继续打
  → 不能 → 看当前读条进度
    → 到滑步窗口了（读条剩余 < 0.45 秒）→ 滑步走人，不浪费当前读条
    → 还没到 → 打断读条立刻走
```

**集合策略（需要跑位集合时用）**：尽快到达，不等。

#### 移动方式

- **寻路移动**：通过 VNavmesh 自动寻路走到目标位置（适合远距离）
- **瞬移**：直接传送到目标坐标（适合紧急情况或来不及走的时候）
- **停留**：原地不动（适合需要站桩的机制）

### ACR：在移动过程中继续输出

移动和输出是**并行**的。智能层负责走路，ACR 负责打技能。

```
角色正在从 A 点走到 B 点（智能层驱动）
  同时：
  ACR 检查 → GCD 好了 → 释放一个瞬发技能（不影响移动）
  ACR 检查 → 能力技好了 → 释放一个能力技（不影响移动）
  ACR 检查 → 需要读条 → 等滑步窗口再读（和移动协调）
```

## 辅助轴：为什么必须独立运行

## 辅助轴：安全坐标计算的核心引擎

辅助轴的根本目的不是"辅助"，而是做**安全坐标计算**——这是实现全自动移动的关键。

为什么必须独立运行？因为安全坐标计算要配合事实轴使用，但执行轴和事实轴是互斥的。如果把坐标计算放在执行轴里，用事实轴模式时就断了；放在事实轴里，无轴模式时又没了。所以它必须独立于模式切换，始终运行。

```
执行轴（模式 A）──┐
                  ├──→  AIRunner  ──→ 技能输出 + 角色移动
事实轴（模式 B）──┤        ↑
                  │        │
辅助轴（始终运行）──┘    始终并行
  └→ 脚本节点计算安全坐标 → 写入移动需求缓冲区
```

辅助轴的脚本节点可以做：安全坐标计算、副本特殊机制响应、全程生效的自定义触发规则。但核心价值是坐标计算——没有它，全自动移动就无法实现。

## 关键设计点

**为什么安全坐标计算放在辅助轴而不是智能层？**

因为坐标计算逻辑是按副本定制的（每个 Boss 的安全区域算法不同），而且要配合事实轴使用。但执行轴和事实轴互斥，所以坐标计算必须独立运行——放在辅助轴里，始终在线，不受模式切换影响。智能层只负责协调：等事实轴的事件到了，释放已经算好的移动需求。

**为什么读条职业要"前向计算"？**

读条职业（黑魔法师、白魔法师等）不能边走边读条。系统会计算："我还能塞一个读条技能再走吗？"如果能，就继续打；如果不能，就在滑步窗口（读条快结束的 0.45 秒内）移动，不浪费当前读条。

**为什么减伤分配用"冷却升序"？**

冷却短的技能更频繁可用。优先用短冷却的减伤，下次需要时它们更可能转好。这是一种贪心策略——用最少的资源满足当前需求，保留长冷却技能应对突发。

**时间表的 Sync 校准是什么？**

Boss 的技能时间表是理论值，实际可能有微小偏差。Sync 校准通过监听游戏中的实际事件（比如 Boss 开始读条），自动修正时间偏差，确保后续事件的时间点准确。

## 常见问题

**Q: 我需要手动配置什么吗？**

不需要。安装插件后，系统会自动识别你的职业、加载对应的战斗方案、检测副本数据。开箱即用。

**Q: 它会帮我走位吗？**

会。这是 HiAuRo 的核心能力之一。辅助轴的脚本节点会根据 Boss 机制计算安全坐标，智能层在合适的时机释放移动需求，移动执行器驱动角色自动走到安全位置。

**Q: 我可以暂停它吗？**

可以。你可以随时暂停（停止按技能但不关闭）或停止（完全关闭自动战斗）。

**Q: 移动过程中还能打技能吗？**

能。瞬发技能在移动过程中正常释放，读条技能会在滑步窗口释放。

**Q: 不同副本的表现一样吗？**

不一样。有预设时间表的高难度副本会使用事实轴模式，表现最好——包括自动走位、自动减伤、自动治疗。没有预设数据的普通副本会使用基础模式，只自动按技能。
