# HiAuRo 自动副本攻略 — 它是怎么帮你打通整个副本的？

## 一句话总结

HiAuRo 是一个战斗辅助框架，服务三层用户：**普通用户**安装即用，**轴作者**为每个副本编写触发树/时间线/坐标计算脚本，**ACR 作者**为每个职业编写技能逻辑。三者配合，实现从自动按技能到全自动副本攻略的完整体验。

**轴作者**：把 Boss 机制翻译成系统能理解的数据

- 为每个副本写事实轴时间线（全队视角的 Boss 技能时间表）
- 为每个副本写辅助轴坐标计算脚本（安全区域在哪里）

**普通用户**：安装即用
- 不需要写任何代码
- 轴作者和 ACR 作者的工作已经封装好了
- 切图时系统自动加载对应的轴数据和 ACR

## 核心思路：两种模式 + 一个常驻层

HiAuRo 有两种战斗模式，互斥使用：

- **执行轴模式**：自己做完了当前副本、当前职业的事实和决策 → **最常用**，一个人就能用
- **事实轴模式**：全队视角的事实+决策，需要全队都用 HiAuRo 才能实现完整的减伤/治疗分配和移动驱动

两种模式都能实现"判断 Boss 做什么 → 决定自己该怎么做"，但方式不同：

```
事实轴：系统提供的完整框架
  → 有预定义的数据模型（时间线、阶段、事件、动作）
  → 全队视角，能分配减伤/治疗
  → 辅助轴算安全坐标，移动执行器驱动角色移动
  → 需要全队都用 HiAuRo

执行轴：灵活的节点编排引擎
  → 没有固定数据模型，ACR 作者自己组合节点实现逻辑
  → 只管当前副本、当前职业
  → 不能智能驱动移动（只能通过复杂计算规划）
  → 不需要全队配合
```

不管用哪种模式，辅助轴始终并行运行（安全坐标计算），智能层统一调度。


## 事实轴和辅助轴：并行合作

### 事实轴：知道 Boss 要做什么

事实轴持有一份 Boss 的技能时间表（提前制作的 JSON 文件），知道：

- 现在是第几阶段（P1、P2、P3...）
- Boss 下一个技能是什么、还有多久放
- 每个技能的机制类型（全屏 AOE？点名？需要分散？需要集合？）

注意：事实轴**不知道安全区域在哪里**。游戏机制是动态的——Boss 朝向、场地状态、队伍站位都在变，不可能提前把坐标写死在 JSON 里。

### 辅助轴：知道你应该往哪跑

辅助轴的脚本节点负责根据**实时游戏状态**计算安全坐标：

- Boss 当前朝向是什么？
- 场地上哪些区域是安全的？
- 你是什么职业？该去哪个位置？
- 队伍怎么分散？

这些计算逻辑写在脚本节点里，按副本编号加载。每个 Boss 的安全区域算法不同，需要单独编写。

### 它们怎么合作？

```
事实轴：Boss 马上要放十字形 AOE（发出事件）
         ↓
辅助轴脚本节点：收到事件通知
  → 读取 Boss 当前朝向（实时）
  → 计算安全坐标：你该去东北角 (10, 0, -15)
  → 生成移动需求，写入缓冲区
         ↓
事实轴：事件继续推进到下一个时间点
```

事实轴提供"什么时候"，辅助轴提供"去哪里"。它们不是谁指挥谁，而是各自做好自己的事，通过事件和数据协作。

## 智能层：统一调度的大脑

智能层包含决策引擎和移动协调，是整个系统的调度中心。

### 统一控制所有 ACR：QT 开关

不同作者写的 ACR，技能逻辑各不相同。HiAuRo 怎么统一控制它们？

答案是 **QT（Quick Toggle）开关**。HiAuRo 定义了一套**全职业通用的 QT 开关**，所有 ACR 作者都必须实现这些开关。系统通过设置这些开关的开/关，就能统一控制所有职业的行为。

```
全职业通用 QT 开关：
  ┌──────────────┬────────────┬─────────────────────────────┐
  │  开关名称     │  默认值    │  作用                        │
  ├──────────────┼────────────┼─────────────────────────────┤
  │  爆发         │  开        │  是否使用爆发技能             │
  │  爆发药       │  关        │  是否使用爆发药               │
  │  停手         │  关        │  是否停止攻击                 │
  │  自动减伤     │  开        │  是否自动释放减伤技能         │
  │  清空资源     │  关        │  是否清空所有资源             │
  │  AOE          │  开        │  是否使用群体攻击             │
  │  TTK          │  关        │  击杀时间模式                 │
  └──────────────┴────────────┴─────────────────────────────┘
```

**谁在控制这些开关？**

- **事实轴**：Boss 进入特定阶段时，自动切换开关（比如"停手"阶段 → 设置 Hold=true）
- **执行轴**：按时间表控制开关（比如第 30 秒开启爆发 → 设置 Burst=true）
- **用户**：通过 UI 手动切换
- **决策引擎**：分配减伤时设置自动减伤开关

**ACR 作者怎么响应？**

ACR 作者在写技能逻辑时，检查对应的 QT 开关：

```
诗人 ACR 的技能选择逻辑：
  if (QTHelper.IsEnabled(BuiltinQt.Hold))  → 停手，不放技能
  if (!QTHelper.IsEnabled(BuiltinQt.Burst)) → 跳过爆发技能
  if (QTHelper.IsEnabled(BuiltinQt.AoE))   → 优先 AOE 技能
  正常选择最优技能...
```

**不同作者写的 ACR，但都检查同一个开关 → 系统就能统一控制所有职业。**

### 事件驱动精调：OnGameEvent 回调

QT 开关解决了"统一控制"的问题。但有些副本需要更精细的控制——比如"Boss 读条某个技能时，提前保留爆发技能"或"被点名时立刻使用位移技能"。

HiAuRo 提供了 `OnGameEvent` 回调，ACR 作者可以在自己的代码里监听游戏事件，实现副本级别的精调：

```
ACR 作者能监听的游戏事件：
  Boss 读条       →  Boss 开始读某个技能，可以提前准备
  Buff/Debuff     →  自己或 Boss 获得/失去某个状态
  连线            →  地图上出现连线机制
  地图特效        →  场地发生变化（安全区/危险区切换）
  点名图标        →  被 Boss 点名
  NPC 喊话        →  Boss 说话暗示下一步机制
  单位创建/消失   →  小怪出现/Boss 消失
  天气变化        →  副本天气变化（某些机制相关）
  技能效果        →  某个技能命中了目标
```

**示例：诗人 ACR 作者针对特定副本的精调**

```csharp
// 在 ACR 的事件回调中
void OnGameEvent(ITriggerCondParams eventParams)
{
    // Boss 读条"审判"→ 保留爆发技能，等机制结束后再用
    if (eventParams is ActorCastParams cast && cast.ActionID == 12345)
    {
        QTHelper.SetValue(BuiltinQt.Burst, false);  // 关爆发
    }
    
    // 被点名"追踪"→ 立刻使用位移技能
    if (eventParams is ActorControlTargetIconParams icon 
        && icon.TargetID == Data.Me.EntityID && icon.IconID == 57)
    {
        SlotHelper.EnqueueSpell(位移技能ID);  // 立刻释放
    }
    
    // Boss 消失 → 停手
    if (eventParams is ActorControlTargetableParams t && !t.IsTargetable)
    {
        QTHelper.SetValue(BuiltinQt.Hold, true);
    }
}
```

**这就是 HiAuRo 的灵活性所在：**

- **QT 开关**解决"统一控制"——所有职业响应同一个开关
- **OnGameEvent**解决"精细控制"——每个 ACR 作者可以针对具体副本写定制逻辑
- 两者配合，既保证了系统级的统一调度，又允许职业级别的个性化优化

### 决策引擎：分配减伤和治疗

事实轴发出"需要减伤"或"需要治疗"的事件后，决策引擎负责分配。

#### 减伤分配：贪心覆盖多次伤害

减伤分配不是简单的"一个事件分配一次"。一个减伤技能有持续时间（比如 15 秒），在这段时间内 Boss 可能放多次伤害。系统会**贪心地让一个减伤尽量覆盖多次伤害**。

```
事实轴时间线：
  第 10 秒：Boss 技能 A（需要 10% 减伤）
  第 15 秒：Boss 技能 B（需要 10% 减伤）
  第 20 秒：Boss 技能 C（需要 10% 减伤）
  第 50 秒：Boss 技能 D（需要 10% 减伤）

骑士的"壁垒"：减伤 15%，持续 15 秒，冷却 90 秒
学者的"鼓舞"：减伤 10%，持续 15 秒，冷却 60 秒

分配策略：
  → 第 10 秒时计算：壁垒在第 10 秒使用，持续到第 25 秒
    → 能覆盖技能 A(10s)、B(15s)、C(20s) 三次伤害！
    → 一个壁垒(15%) 就够了，不需要再分配鼓舞
  → 第 50 秒时计算：壁垒还在冷却，用鼓舞(10%)覆盖技能 D
```

**关键约束：只看附近的伤害事件。** 计算时只考虑一个时间窗口内的伤害，不能把所有未来的伤害都算进来。否则所有减伤都会被用来"覆盖多次伤害"，后面的伤害就没减伤可用了。

```
正确：第 10 秒计算时，只看 25 秒以内的伤害事件
错误：第 10 秒计算时，把整场战斗的伤害都算进来
```

#### 治疗分配：立即执行

治疗需求更紧急——会**立即执行**，不等 deadline。

#### 分配原则

- **冷却短的优先用**：这样下次需要时更可能转好
- **持续时间长的优先覆盖多次**：贪心最大化减伤利用率
- **不足时标记**：如果队伍的减伤/治疗不够满足需求，会标记"不足"

### 移动协调：释放移动需求，驱动角色移动

辅助轴的脚本节点算好坐标后，写入移动需求缓冲区。智能层负责：

1. **监控事实轴事件** — 等事件到了再释放（不到时候不走）
2. **释放移动需求** — 把缓冲区里的需求激活
3. **驱动移动执行器** — 通过寻路插件（VNavmesh）走到目标位置

#### 两种移动策略

**惰性策略（打 Boss 机制时用）**：尽量晚走，多打一个技能再走。

```
读条职业的决策过程：
  → 还能塞一个读条技能吗？
  → 能 → 继续打
  → 不能 → 看当前读条进度
    → 到滑步窗口了（读条剩余 < 0.45 秒）→ 滑步走人
    → 还没到 → 打断读条立刻走
```

**集合策略（需要跑位集合时用）**：尽快到达，不等。

#### 移动方式

- **寻路移动**：通过 VNavmesh 自动寻路走到目标位置（适合远距离）
- **瞬移**：直接传送到目标坐标（适合紧急情况）
- **停留**：原地不动（适合需要站桩的机制）

### 移动和输出是并行的

智能层负责走路，ACR 负责打技能。它们同时运行：

```
角色正在从 A 点走到 B 点（移动执行器驱动）
  同时：
  ACR 检查 → GCD 好了 → 释放一个瞬发技能（不影响移动）
  ACR 检查 → 能力技好了 → 释放一个能力技（不影响移动）
  ACR 检查 → 需要读条 → 等滑步窗口再读（和移动协调）
```

## 辅助轴为什么必须独立运行

事实轴和辅助轴是并行的，但执行轴和事实轴是**互斥**的——同一时间只能用一个。

如果把辅助轴的坐标计算放在执行轴里，切到事实轴模式时就断了。放在事实轴里，无轴模式时又没了。所以辅助轴必须独立于模式切换，始终运行。

```
执行轴（模式 A）──┐
                  ├──→  AIRunner  ──→ 技能输出 + 角色移动
事实轴（模式 B）──┤        ↑
                  │        │
辅助轴（始终运行）──┘    始终并行
```

## 执行轴：节点编排引擎

执行轴是事实轴的**替代方案**——它们互斥，同一时间只能用一个。

执行轴不是一个完整的事实+决策系统，而是一个**节点编排引擎**。它提供了一套灵活的节点（条件、动作、延迟、脚本等），ACR 作者通过编排这些节点来实现和事实轴同样的效果——判断 Boss 在做什么、决定自己该怎么做。

关键区别：
- **事实轴**：系统提供的完整框架，有预定义的数据模型（时间线、阶段、事件、动作）
- **执行轴**：灵活的编排工具，没有固定的数据模型，ACR 作者自己组合节点来实现逻辑

执行轴只管**当前副本、当前职业**。它不能智能驱动移动（只能通过复杂计算规划移动路径），不需要全队配合。

### 触发树：执行轴的核心

执行轴用一棵**触发树**来表达事实判断和决策逻辑，用 JSON 文件定义，按副本编号存放。树的每个节点是一个"条件+动作"的组合：

```
触发树示例：
  根节点
  ├── [条件] 战斗开始后 3 秒
  │   └── [动作] 开启爆发
  ├── [条件] 战斗开始后 10 秒
  │   └── [动作] 释放极限技
  ├── [条件] Boss 读条"审判"
  │   ├── [动作] 切换到停手模式
  │   └── [延迟] 等 5 秒
  │       └── [动作] 切换回正常模式
  └── [条件] Boss 血量低于 30%
      └── [动作] 开启爆发 + 清空资源
```

### 触发树能做什么？

- **强制释放技能** — 直接让 ACR 使用指定技能，跳过正常选择逻辑
- **暂停/恢复 ACR** — 在特定阶段停止自动战斗
- **切换目标** — 强制切换到指定目标
- **控制 QT 开关** — 开启/关闭爆发、停手、AOE 等开关
- **等待条件** — 挂起直到某个条件满足（比如 Boss 读条结束）
- **延迟执行** — 等待指定时间后执行下一步
- **运行脚本** — 执行自定义 C# 脚本做复杂判断
- **规划移动** — 通过复杂计算规划移动路径（不能智能驱动，只能规划）

### 执行轴和事实轴怎么选？

```
大部分情况（只有你用 HiAuRo）→ 用执行轴或无轴模式
  → 执行轴：自己做这个副本、这个职业的事实+决策（爆发/停手/AOE 切换等）
  → 无轴：没有预设数据时，ACR 自己根据当前情况选择技能
  → 这是最常见的使用场景

理想情况（全队都用 HiAuRo）→ 用事实轴模式
  → 全队视角的事实+决策，智能分配减伤/治疗
  → 辅助轴算安全坐标，移动执行器驱动角色移动
  → 需要全队配合，单人无法实现完整的减伤/治疗分配和移动驱动
```

切图时系统会自动检查有没有对应的 JSON 文件，按配置优先级选择模式。

## 关键设计点

**为什么事实轴和辅助轴要分开？**

因为它们解决不同的问题。事实轴是"Boss 要做什么"（静态时间表），辅助轴是"你应该往哪跑"（动态计算）。游戏机制是动态的，安全坐标不可能写死在 JSON 里，必须实时计算。它们并行运行，各自做好自己的事，通过事件协作。

**为什么安全坐标计算放在辅助轴？**

因为坐标计算要配合事实轴使用，但执行轴和事实轴互斥。如果放在执行轴里，切到事实轴模式时就断了。所以必须独立运行。

**为什么读条职业要"前向计算"？**

读条职业（黑魔法师、白魔法师等）不能边走边读条。系统会计算："我还能塞一个读条技能再走吗？"如果能，就继续打；如果不能，就在滑步窗口（读条快结束的 0.45 秒内）移动，不浪费当前读条。

**为什么减伤分配用贪心覆盖策略？**

一个减伤技能有持续时间（比如 15 秒），这段时间内 Boss 可能放多次伤害。如果每次都单独分配减伤，会浪费资源。贪心策略让一个减伤尽量覆盖多次伤害，最大化利用率。但必须限制在时间窗口内计算——不能把整场战斗的伤害都算进来，否则所有减伤都会被"覆盖多次伤害"消耗掉，后面的伤害就没减伤可用了。

**时间表的 Sync 校准是什么？**

Boss 的技能时间表是理论值，实际可能有微小偏差。Sync 校准通过监听游戏中的实际事件（比如 Boss 开始读条），自动修正时间偏差，确保后续事件的时间点准确。

## 常见问题

**Q: 我需要手动配置什么吗？**

不需要。安装插件后，系统会自动识别你的职业、加载对应的战斗方案、检测副本数据。开箱即用。

**Q: 它会帮我走位吗？**

会。这是 HiAuRo 的核心能力之一。辅助轴的脚本节点会根据 Boss 机制和实时游戏状态计算安全坐标，智能层在合适的时机释放移动需求，移动执行器驱动角色自动走到安全位置。

**Q: 我可以暂停它吗？**

可以。你可以随时暂停（停止按技能但不关闭）或停止（完全关闭自动战斗）。

**Q: 移动过程中还能打技能吗？**

能。瞬发技能在移动过程中正常释放，读条技能会在滑步窗口释放。

**Q: 不同副本的表现一样吗？**

不一样。有预设时间表和坐标计算脚本的高难度副本会使用事实轴+辅助轴模式，表现最好——包括自动走位、自动减伤、自动治疗。没有预设数据的普通副本会使用基础模式，只自动按技能。
